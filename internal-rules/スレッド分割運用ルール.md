# スレッド分割運用ルール

## 目的

本部、運転、監査、PJ ごとの文脈を分離し、長時間運転でも文脈汚染を起こしにくい形にする。

## 基本原則

- スレッドは `本部`, `運転`, `監査`, `学術`, `人事`, `内製開発`, `PJ別` に分ける。
- 各スレッドは自分の bounded work を優先し、担当外の構造変更は本部へ戻す。
- 本部スレッドは方針決定、制度変更、例外判断、優先順位再編だけを担う。
- 運転スレッドは会議、台帳、CLI 補助実装、定常前進を担う。
- 監査スレッドは script audit、境界逸脱、形骸化確認を独立に担う。
- PJ スレッドは各事業の triad 運営と bounded improvement を担う。

## 戻し条件

- high risk の新規開始
- 憲法 / 法律 / 定款改定
- 鍵、秘密情報、不可逆操作
- PJ 横断の優先順位衝突
- 既存部署では吸収せず、新部門や新部署が必要な場合

## 本部の役割

- active / pending / dormant の全体配列を見る
- 次にどの束を前へ出すか決める
- スレッド間 handoff の詰まりを取る
- 必要に応じて部署や部門の拡張審議を開く
