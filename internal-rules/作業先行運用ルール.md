# 作業先行運用ルール

## 目的

議題だけが増え続けて実作業が進まない状態を防ぎ、組織の主出力を `会議消化` ではなく `実装前進` に固定する。

## 基本原則

- 普通の運転では `会議 1` に対して `実作業 2` 以上の前進を要求する
- 新規議題は、既存の active 実装束か active PJ の前進を妨げない範囲でしか開かない
- 実装前進がない sitting は原則 `report-only` に落とし、discussion を増やさない

## 会議予算

- ordinary sitting で新しく `discussion` に上げられる substantive item は最大 `2` 件
- 直近サイクルで実装前進が `0` の場合、新規 discussion item は原則 `0` 件
- `report` は watch 更新だけに限定し、追加資料の量を増やしすぎない

## 作業前進の定義

- rule, SOP, ledger, template を実装系部署が運用可能な状態まで更新した
- 自動化束の state が `queued -> active`, `active -> watch`, `watch -> rolled-forward` のいずれかへ進んだ
- active PJ または active bundle に対し、受入可能な bounded artifact を1つ以上残した

## 議題凍結

- `2` cycle 連続で実装前進がない owner の新規議題は凍結する
- 凍結解除には、先に実作業か watch 消化を完了する
- founder-office が必要と認めた例外だけ凍結を外せる
