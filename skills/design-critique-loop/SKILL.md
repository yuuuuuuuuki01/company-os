# Design Critique Loop

Use this skill to turn vague dissatisfaction like "the UI is bad" into a repeatable design-improvement loop.

## Core workflow

1. Diagnose the current UI in one sentence.
2. Score it quickly on clarity, hierarchy, density, contrast, motion, and fit to product intent.
3. Name one primary failure mode.
4. Choose one visual direction before changing anything.
5. Limit the next pass to one or two strong changes, not a total rewrite.
6. Review the result against the same scorecard.
7. Record the lesson so the next pass starts stronger.

## Direction rules

- Pick a point of view, not a neutral average.
- Match the visual language to the product's role.
- Prefer stronger hierarchy before adding decoration.
- Reduce "feature soup" before adding more components.
- Use metaphor carefully; keep state and action readable first.

## Critique rules

- Critique the artifact, not the taste of the reviewer.
- Name the evidence in the screen that caused the judgment.
- Prefer "too flat", "too noisy", "weak grouping", or "unclear action path" over generic words like "bad".
- Always propose the next experiment.

## Training loop

- Read [references/scorecard.md](references/scorecard.md) for the scoring rubric.
- Read [references/drills.md](references/drills.md) when running practice rounds.

## Output contract

When using this skill, produce:

- a one-line diagnosis
- the top one or two failure modes
- one chosen visual direction
- one short next-pass plan
- one reusable lesson for the organization
